# Tinyfish-only vs Tavily-only：2026-07-17 纳斯达克报告对比

## 测试设置

- 问题：`调查2026-07-17的纳斯达克指数`
- 报告类型：`research_report`
- 语气：`objective`
- 数据源：`web`
- 除 `RETRIEVER` 外，其余 `.env` 中的 LLM、embedding、语言和字数配置相同
- 两轮均禁用 PDF 与 DOCX，避免格式转换影响耗时
- Tinyfish 轮：`RETRIEVER=tinyfish`
- Tavily 轮：`RETRIEVER=tavily`

| --- | ---: | ---: |
| 总耗时 | 171.07 秒 | 243.45 秒 |
| 报告总成本 | $0.304434 | $0.463631 |
| 报告字符数 | 15,812 | 12,710 |
| 唯一引用 URL | 7 | 14 |
| 唯一引用域名 | 5 | 8 |
| 检索阶段可见候选 URL | 有，两轮检索均返回结果 | 本次完整运行中为 0 |

本次样本中，Tinyfish 比 Tavily 快约 29.7%，总成本低约 34.3%，报告篇幅长约 24.4%。

## 关键事实核验

本题存在一个容易被忽略的时区问题：中文媒体在北京时间 2026-07-17 发布的“美股收盘”报道，往往对应美国东部时间 2026-07-16 的交易。

美东时间 2026-07-17 的日终数据为：

- 纳斯达克综合指数收于 **25,520.24**
- 下跌 **361.70 点**
- 跌幅约 **1.4%**

对比两份报告：

- Tinyfish 报告明确承认没有取得准确收盘点数，但把 **-1.5%** 当作 7 月 17 日表现；该数字实际对应 7 月 16 日。
- Tavily 报告给出 **25,881.95、-387.28、-1.47%**，这些数字也明确对应 7 月 16 日，而非 7 月 17 日。

因此，两份报告都没有正确回答“美国交易日 2026-07-17”的核心行情问题。Tavily 的数字看起来更完整，但日期错配使其精确度具有误导性。

## 内容质量观察

### Tinyfish-only

优点：

- 在线 smoke test 和完整检索均能稳定返回 URL。
- 完整运行明显更快、更便宜。
- 对无法确认的收盘点数有一定不确定性表达。
- 找到了 Nasdaq、FRED、Reuters 等潜在高质量候选来源。

缺点：

- 最终报告只使用了 7 个唯一 URL，引用集中度较高。
- 若干候选页面正文抓取失败，导致搜索覆盖没有充分转化为证据。
- 混入低相关来源，并对有限证据做了过强的宏观归因和投资判断。
- 未识别中文发布时间与美国交易日之间的时区错配。

### Tavily-only

优点：

- 最终文本结构清楚，行情表格和技术分析更具体。
- 最终报告列出的 URL 和域名数量更多。

缺点：

- 本次完整运行的 6 个检索请求均显示 0 个进入抓取阶段的 URL；运行后单独调用 Tavily API 又能返回结果，说明可能存在并发、瞬时服务状态或流水线过滤问题。
- 在缺少检索上下文的情况下，写作阶段仍生成了大量精确数字和引用，可信度风险很高。
- 把 7 月 16 日的 25,881.95 / -1.47% 错标为 7 月 17 日。
- 耗时和成本均显著高于 Tinyfish。

## 结论

在这一次真实运行中，**Tinyfish 更适合作为 Tavily 的替代检索器**：它能在完整流水线里实际返回来源，而且速度和成本均更好。但当前不能据此认定 Tinyfish 生成的报告更准确；两份报告都被交易日与中文报道日期的错配误导。

建议默认使用：

```bash
RETRIEVER=tinyfish
```

如继续使用混合检索：

```bash
RETRIEVER=tinyfish,codex
```

金融日期型任务还应在查询中明确加入“美国东部时间交易日”，并在写作前增加一个硬性行情核验步骤：至少用一个日终指数数据源确认日期、收盘点位和涨跌幅后，才允许生成归因与展望。
